-----------------------------------
-+- The Console and the Parse -+-
-----------------------------------

In game there is a console which is used for entering commands and debugging any game.
Use the tilde key on your keyboard to bring it down, and the HELP command for see the
list of commands it accepts. A "commands.ini" file exists in the game's directory and
it is used to run commands immediately when the game is loaded only accepting commands.
There are also .PX files which exist in the \Levels folder under the game's directory.
These files are for setting up data object models and do accept commands started with
an exclamation mark denoting that it should be considered a console command executed.

One of the first things to note that the parser recognizes is comments starting with ;
The parser accepts a data model script at the initial occurrence level *with exceptions*
Upon running the game, a file "commands.ini" is executed as a list of console commands,
the LEVEL command is used which loads the supplied .PX title from the levels directory.
There is also a PARSE console command it does not reset any of the level requirements.
With in a PX file you may also include a PARSE tag which will include another PX script.
The following is an example of using ways to embed nested PARSE abilities in a PX file.

parse
{
     filename <title>
     ;will load <game path>\levels\<title>.px
}

All tags are similar for the data model in that they all start with their respected name
and are followed by the enclosing brackets {} that encase the sub statements for the tag.
Instructional control statements also exist such as methods/variables/if/elseif and else.
Preparation statements are what build an actual media/art data experience into something.
For extendibility these types of statements do have unique arrangements that may affect it.
For instance a beacon tag can be used for every location to place a beacon but at the same
respect the beacon tag allows the sub statement of location to be repeated to multiply it.

----------------------------
-+- Special Characters -+-
----------------------------

Exclamation "!"

       -Any statement with a ! in front of it is interpreted as a console command immediately following.

Ampersand "&"

       -Any statement with a & in front of it is assumed as a method identity immediately following.
       This is not similar to an inclusion of a method, as it would evaluate expressions on pass.

Dollar sign and Period "$ ."

       -Any statement with a $ in front of it is assumed as a variable identity immediately following then
       immediately followed by a . is the property of the variable. Variables created in script all have
       a "value" property. Variables or identities of data model object have various properties relating
       to the sub statements and values that set up the object.

Equals sign "="

       -Any statement with = in front of it evaluates any variable with $ after it as a means to create a
       statement that sets the variable to the expression it equals at the time. Example and use is for
       in the database tag when in serialization you want the variable to re-execute in de-serialization
       storing the value that is in the variable during serialization. So in serialize you would include
       =$variable.value
       Saying that the value is false for instance. The next line as seen in of code would store a false
       $variable.value false
       During de-serialization the pallet plus any additional execution commands in de-serialize would run.

Named Variables

       -There are two named variable identities that are allowed text to be used in the game, those are;
       idletext, and credittext
       Each displays text, idletext is for when the character is not moving and also the entry of the game,
       Credittext is displayed at the end of the game, both these must be in double quotes and may contain
       \n for new line character.

-+- If/Elseif/Else Statements -+-
-----------------------------------

       If statements assume evaluating a Boolean unless an "is" statement is coupled with an if or elseif.
       Else alone is optional as it were. Expressions evaluated are expressed in lateral they are evaluated
       accumulative, this means that any not statement expresses the immediate value after, also any "and"
       or "or" append to the running expression. Below are examples of syntax of If, Elseif, and Else.

       if (exp or not exp) is (value) [
       ] elseif (not exp and exp) is (ok) [
       ] elseif ($id.value) [
       ] else [
